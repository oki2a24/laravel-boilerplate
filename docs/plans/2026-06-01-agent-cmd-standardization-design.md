# 設計書: エージェント実行プロセスの標準化 — ガードレール・アーキテクチャの構築

## 日付: 2026-06-01

## 1. 目的
本設計の目的は、エージェントがプロジェクトのコンテキスト（Docker/Makefile）を無視して、ホスト側のシェルで直接コマンド（`php`, `composer`, `npm` 等）を実行してしまうエラーを根絶することである。単なる「操作手順の周知」ではなく、エージェントの指示層（Instruction Layer）に対して、回避不可能な「ガードレール（防護柵）」を構築することを目指す。

## 2. 課題と根本原因
*   **課題**: エージェントが、慣習的な標準コマンド（例: `php artisan ...`）を、プロジェクト固有の環境（Dockerコンテナ内）を考慮せずに、ホスト側のプロセスとして実行してしまう。
*   **根本原因**: エージェントがプロジェクトの「自律的な実行ルール」を、自身の「推論（Reasoning）プロセス」の最上位レイヤーに組み込めていない。既存の `Makefile` や `AGENTS.md` が、「参照すべき知識」としては存在していても、「実行を制限する命令（Constraint）」として十分に作用していない。

## 3. 解決策: ガードレール・アーキテクチャ
本設計では、エージェントの意思決定プロセスに、以下の3つのレイヤーからなるガードレールを導入する。

### 3.1. 指示層の強化 (Contextual Guardrail)
エージェントがプロジェクトのコンテキストを読み込む際、単なるドキュメントとしてではなく、**「実行プロトコル」**として認識させる仕組みを構築する。

*   **`AGENTS.md` への命令統合**: プロジェクトの `AGENTS.md`（またはエージェントのシステム指示層）に対して、「実行の優先順位（Hierarchy of Execution）」を、実行を制約する**「厳格な制約条件（Hard Constraint）」**として定義・注入する。
*   **実行階層モデルの定義**:
    1.  **Layer 1 (Match)**: `Makefile` 内のターゲット検索。
    2.  **Layer 2 (Compose)**: `docker compose exec` によるコンテナ内実行。
    3.  **Layer 3 (Forbidden)**: ホスト側直接実行の禁止。

### 3.2. 実行経路の単一化 (Single Path Enforcement)
エージェントがコマンドを生成する際、中間層として `Makefile` または `docker compose` を必ず経由するように、思考の「経路」を制限する。

*   エージェントの思考プロセスにおいて、「コマンド生成 → 実行」の間に、**「`Makefile` ターゲットへの再マッピング」**というステップを強制的に挿入する。

### 3.3. 自己修正ループの確立 (Self-Correction Loop)
万が一、ガードレールをすり抜けてホスト側で実行が試行され、失敗（例: `command not found`）した場合の、自律的なリカバリ・メカニズムを定義する。

*   **検知**: ホスト側でのコマンド実行失敗を、環境の不一致（Violation）として認識。
*   **再評価**: `Makefile` の内容を再スキャンし、正しい実行ターゲットを再特定。
*   **リトライ**: `make` または `docker compose exec` を用いた正しい経路での再実行。

## 4. 成功の定義
*   **定量的指標**: エージェントの実行ログにおいて、ホスト側での直接的な `php`, `composer`, `npm` 等のプロセス起動回数が 0 になること。
*   **定性的指標**: エージェントが、新しい `Makefile` ターゲットが追加された際、事前の指示なしに「自律的に」そのターゲットを検出し、使用できること。

## 5. 今後の検討事項
*   `AGENTS.md` の更新に伴う、エージェントの「命令の重み付け」の最適化。
*   より複雑な依存関係（例: `npm` 実行時の `node_modules` の状態確認）に対するガードレールの拡張性。
